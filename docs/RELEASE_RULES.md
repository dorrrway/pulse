# 发布与文档规则

最后更新：2026 年 5 月 13 日

本文件是 Pulse 的发布前检查规则。以后任何代码、文档、安装包或 GitHub 发布相关工作，都必须先按这里检查。

## 文档必须同步的情况

以下任一情况发生时，不能只改代码，必须同时检查并按需更新 README、英文 README、隐私协议和更新日志：

- 新增、删除或改变用户可见功能。
- 改变菜单栏面板、设置项、安装方式、开机启动、运行方式或本地安装脚本。
- 改变读取的数据类型，例如 CPU、内存、磁盘、网络、电池、热状态、进程名称、进程资源占用等。
- 改变数据存储、上传、联网、权限、沙盒、签名、notarization 或登录项语义。
- 改变发布产物，例如版本号、DMG、校验值、下载链接、官网链接或最低系统要求。
- 修复会影响用户判断的问题，例如隐私边界、指标含义、排行口径、单位、采样频率或展示规则。

## README 规则

- 中文 README 是主要入口，英文 README 必须保持等价信息。
- README 必须至少覆盖：下载、安装、系统要求、隐私摘要、更新日志和开发验证命令。
- 如果截图或预览图不能反映当前主要界面，发布前要更新图片；如果暂不更新，要在最终说明里明确指出。
- 下载链接、版本号、DMG 文件名、SHA-256、Developer ID 和 notarization 文案必须和实际发布产物一致。

## 隐私规则

涉及以下内容时，隐私协议必须更新日期和数据清单：

- 新增读取本机数据、系统指标、进程信息、文件路径、设备信息或用户偏好。
- 新增本地存储、网络请求、上传、遥测、日志收集或第三方服务。
- 改变 sandbox、entitlements、登录项或后台运行语义。

隐私文案必须清楚说明：

- 读取了什么。
- 为什么读取。
- 是否存储。
- 是否上传。
- 是否包含个人文件、凭据、钥匙串、定位、剪贴板或持久追踪标识。

如果只在本机界面展示，也要明确写出“不上传”。

## 更新日志规则

- 每次用户可见功能、行为、隐私边界、安装方式或发布产物变化，都要更新 README 的更新日志。
- README 更新日志必须面向用户，合并重复项，不记录内部实现细节、按钮尺寸、文件格式、临时修复路径或只对维护者有意义的过程信息；这类内容写入 `docs/CHANGELOG_INTERNAL.md`。
- 未正式打 tag 或生成新安装包前，变化写入 `未发布` / `Unreleased`。
- `未发布` / `Unreleased` 标题不写日期；真正发布版本时，再把其中内容移动到 `版本号 - YYYY-MM-DD` / `Version - YYYY-MM-DD` 标题下。
- 中文 README 和英文 README 的版本标题、发布日期、变更范围必须保持一致。
- 发布包生成后，对应版本说明必须补充下载链接和 SHA-256 校验值；GitHub README 上的发布说明也必须使用相同版本号和发布日期。
- 纯内部重构、测试补充或无用户可见变化的文档维护，可以不写更新日志，但最终说明要说明判断理由。

## 发布前验证

每次发布前至少执行：

```sh
git status --short --branch
git diff --stat
git diff --check
xcodebuild test -project pulse.xcodeproj -scheme pulse -destination 'platform=macOS'
```

如果涉及安装、登录项、菜单栏运行、签名、权限、沙盒或 Release 行为，还必须执行：

```sh
./scripts/install-local-app.sh
codesign --verify --deep --strict --verbose=2 ~/Applications/Pulse.app
pgrep -afil 'Pulse|pulse'
```

如果只是文档变更，可以只执行 `git diff --check`，但提交说明里要明确这是文档变更。

## Git 发布规则

- 发布前必须 `git fetch origin main`，确认本地分支没有落后远端。
- 工作区混有无关改动时，禁止 `git add -A`。
- 只能显式暂存本次范围内的文件。
- 本地机器状态、Xcode 用户数据、DerivedData、临时日志、截图草稿、未确认实验改动不得进入提交。
- 提交信息要短而明确，描述最终用户或维护者能理解的变化。
- 推送后必须确认 `HEAD` 与 `origin/main` 一致。

## 最终说明规则

发布完成后的回复必须包含：

- 提交 SHA。
- 推送分支。
- 变更范围。
- 实际执行的验证。
- 是否更新 README、英文 README、隐私协议和更新日志。
- 如有未提交或未纳入范围的本地改动，必须明确列出。

## 判断优先级

当规则冲突或不确定时，按以下优先级处理：

1. 用户隐私和数据边界。
2. 安装包、签名、notarization、下载链接和 SHA-256 的真实性。
3. 用户可见文案和界面实际行为一致。
4. 更新日志覆盖用户能感知的变化。
5. 提交范围干净，不混入无关改动。
